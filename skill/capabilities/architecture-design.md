# Architecture Design Capability

## Purpose

Turn accepted product requirements and repository evidence into explicit technical decisions.

## Procedure

1. Define system context and technical boundaries.
2. Select technology from product, Studio OS delivery and support capability, operational, lifecycle-cost, and existing-project evidence.
3. Design components, data ownership, interfaces, and integrations.
4. Select applicable engineering and domain standards.
5. Address security, privacy, reliability, observability, deployment, and migration when relevant.
6. Record consequential decisions as ADRs.
7. Estimate delivery using ranges, assumptions, dependencies, confidence, standards gates, and risk.
8. Prefer existing project patterns unless an accepted decision justifies change.

## Evidence

- Architecture document tied to roadmap capabilities.
- Technology decision with alternatives and studio-owned delivery and support evidence.
- Project Standards Profile.
- ADRs for material decisions.
- Delivery estimate with explicit assumptions.
- Risks and Interface Design or Development handoff according to the selected workflow.
